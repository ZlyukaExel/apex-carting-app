# Функциональные требования

Приоритеты по MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии времени; **Could** — приятное дополнение; **Won't** — вне границ скоупа.

О нумерации требований. ID требований (FR-NN) сохраняются стабильными для трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре).

## Границы скоупа (клиентское приложение + API)

Скоуп — **клиентское мобильное приложение и API для него**. Функции маршала, владельца и
администратора обеспечиваются **существующей инфраструктурой** и здесь не описываются.

Следствия для скоупа:

- **Слоты (эвенты) приходят из существующего бэкенда через API.** Экранов создания и редактирования расписания в клиентском приложении нет.
- **Конфигурации трасс и маршалы** доступны клиенту как справочные (read-only) данные из API — клиент видит их в карточке слота, но не управляет ими.

Семантика фильтров (FR-38). Внутри одной группы значения объединяются по **ИЛИ (OR)**:
`track_type` и `marshal_id` — массивы значений (например, выбраны и новичковая, и опытная
трасса → показать обе). Между разными группами фильтров действует **И (AND)** (например, тип
трассы И период дат И маршал). Границы периода дат — **включительные**; даты вводятся в
локальной дате клиента. По умолчанию фильтры не заданы (показывается дефолтная выдача — слоты
на ближайшие 7 дней, см. FR-9 и R-027). Если справочник маршалов пуст, фильтр по маршалу скрывается. Активные
(применённые) фильтры отображаются клиенту.

Граница со серверной интеграцией. Детали серверной реализации брони (транзакции, SLA,
маппинг полей) — вне скоупа клиентского приложения. Существующий бэкенд выступает black-box
источником истины; атомарность операций и гарантия «0 двойных броней» обеспечиваются на его
стороне. Граница между приложением и бэкендом закрывается контрактом API; клиент полагается на
коды ответов и значения полей из ответа (включая `max_karts`, `free_rental_equipment`,
`price_total`).

Данные существующей инфраструктуры (R-015). Проект учебный/тестовый, легаси-данных нет.
Каноническая схема — **контракт API**: бэкенд по условию предоставляет все
поля контракта (`rental_price`, `geometry`, `meeting_point`, `free_rental_equipment` и пр.).
Требований к миграции, backfill и дозаполнению исторических слотов не формулируем — это вне
скоупа клиентского приложения.

## Модель цены (учёт)

Итоговая стоимость брони рассчитывается сервером по модели:
`price_total = price × seats_count + rental_price × rental_count`,
где `price` — цена места на слоте, `seats_count` — число мест в брони, `rental_price` — цена
проката одного комплекта экипировки, `rental_count` — число прокатных комплектов. Своя экипировка бесплатна; прокат
доплачивается отдельным тарифом. Валюта — **RUB**. Цена фиксируется на момент создания брони
и далее не меняется. Приложение отображает итог через `price_total` из ответа API (см. FR-45),
не пересчитывая его локально.

поздняя / отменена центром). Граница ранней отмены — ровно за 2 часа до старта включительно
(см. FR-17).

Вне скоупа. Функции маршала, владельца/администратора, расписание-CRUD, онлайн-оплата,
авто-погода и программа лояльности обеспечиваются существующей инфраструктурой или относятся к
развитию — в клиентском приложении и его API не реализуются.

Оценки маршалов — Phase 2 (осознанное сужение скоупа, подтверждено заказчиком). Оценки
маршалов **не входят в MVP** и переносятся в Phase 2. Это сознательное решение по сужению
скоупа: заказчик ранее называл эту функцию критичной, поэтому она не отбрасывается, а планируется
к реализации в Phase 2 — после запуска критичного набора MVP (просмотр слотов, запись, отмены) в
жёсткий срок начала сезона. **Перенос согласован с заказчиком.** На MVP-скоуп и
трассировку требований это не влияет.

«Поделиться» заездом — Won't (MVP) / Phase 2 (согласовано с заказчиком). Функция «поделиться»
слотом (share заезда: deep-link/текст заезда) **не входит в MVP** и переносится в Phase 2. На
карточке заезда иконка «Поделиться» в MVP неактивна/декоративна; специфицировать
поведение шеринга в MVP не требуется. **Согласовано с заказчиком.** На
MVP-скоуп и трассировку требований это не влияет.
