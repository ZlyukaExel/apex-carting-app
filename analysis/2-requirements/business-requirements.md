# Бизнес-требования

Скоуп — **клиентское мобильное приложение и API для него.** Бизнес-цели, обслуживаемые
существующей инфраструктурой (картина для владельца, интерфейс маршала, оценки), в этот
список не входят.

О нумерации требований. ID требований (BR-NN, M-NN) сохраняются стабильными для
трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее
заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре).

Роли в текущей поставке. В текущую поставку входит только роль **Клиент** (самостоятельная
запись на заезды с мобильного приложения). Интерфейсы маршала и администратора/владельца
— **вне скоупа текущей поставки**: они обеспечиваются существующей инфраструктурой.

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| BR-1 | Устранить ручную путаницу и двойные брони, возникающие при ведении записи через Telegram и доску маркером. Учёт записей должен вестись в единой системе, исключающей конфликты по картам. | Must | Бриф (письмо Дениса, п. о хаосе и двойных записях) |
| BR-2 | Дать клиентам возможность самостоятельно записываться на заезды онлайн без ручного посредничества владельца («люди сами записываются, а я только смотрю»). | Must | Бриф (письмо Дениса) |
| BR-5 | Снизить простой картов из-за поздних отмен за счёт правила отмены 2 часов и фиксации поздних отмен (контроль без отталкивания клиентов — «ограничить, но не злить»). | Must | Бриф (карт простаивает при отмене за 10 минут); Ответы заказчика № 7–8 |
| BR-7 | Запустить продукт к старту высокого сезона (жёсткий срок ~2 месяца) при ограниченном бюджете, начав с минимально необходимого набора функций (MVP-подход). | Must | Бриф (сроки и бюджет); Ответы заказчика № 20 |
| BR-8 | На старте оплата остаётся офлайн (наличные / перевод); продукт лишь фиксирует запись и показывает цену. Онлайн-оплата — за рамками скоупа, заложить в развитие. | Must | Бриф (наличка/перевод, не уверены в онлайн-оплате); Ответы заказчика № 9, 11 |
| BR-9 | Клиентское приложение — мобильное (использование у трассы с телефона); взаимодействует с существующим бэкендом через API. | Must | Бриф (пользуются с телефона); Ответы заказчика № 19 |

## Метрики успеха

Целевые значения — разумные ориентиры для MVP (точные пороги фиксируются как допущение, при наличии данных уточняются).

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :-- | :-- | :-- | :-- |
| M-1 | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей за сезон). | ≥ 70% к концу первого сезона | BR-2 |
| M-2 | Число двойных броней / конфликтов по картам. | 0 (полное устранение) | BR-1 |
| M-4 | Простой картов из-за поздних отмен (доля поздних отмен от записанных). | Снижение относительно текущего уровня (целевой ориентир ≤ 10%) | BR-5 |
| M-6 | Готовность к старту сезона. | Запуск в срок ~2 месяцев с критичным набором функций (просмотр слотов, запись, отмены) | BR-7 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало высокого сезона (~2 месяца); бюджет ограничен → строго MVP-подход, «навороты» откладываются. (Бриф; Ответы заказчика № 20)
- **Оплата офлайн на старте.** Онлайн-оплата вне границ скоупа. (Ответы заказчика № 9, 11)
- **Mobile-first.** Клиентское приложение — с телефона; данные получает из существующего бэкенда через API. (Ответы заказчика № 19)
- **Вне скоупа.** Функции маршала/владельца, расписание-CRUD и админка обеспечиваются существующей инфраструктурой. Онлайн-оплата, автоматический учёт погоды, программа лояльности, публичный рейтинг — направления развития. (Описание домена → «Границы скоупа»)
- **Оценки маршалов — Phase 2 (подтверждено заказчиком).** Оценки маршалов осознанно вынесены в Phase 2 и не входят в MVP. Заказчик ранее называл функцию критичной, поэтому она не отбрасывается, а планируется к реализации после запуска критичного набора MVP (BR-7) — это сознательное сужение скоупа ради попадания в жёсткий срок начала сезона. **Перенос согласован с заказчиком.** (Бриф; Описание домена → «Границы скоупа»; решение R-001)
